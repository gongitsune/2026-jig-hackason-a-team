# 状態遷移・同期 実装ガイド

**制約: 7時間以内の簡易実装** を前提に、ポーリングによる状態同期の実装方針をまとめる。

---

## 1. 全体方針

| 項目 | 方針 | 理由 |
|------|------|------|
| 同期方式 | GET /rooms のポーリングのみ | WebSocketなし、実装が単純 |
| ポーリング間隔 | **2秒固定** | バックオフは省略、今日中に動かす優先 |
| 状態の真実 | バックエンドのDB | フロントは常にGETで最新を取得 |
| 画面遷移 | status + round で判定 | クライアントは表示する画面を決めるだけ |

---

## 2. ルーム状態(status)の定義

```
WAITING        → 開始待ち（start.html）
WORD_INPUT     → 単語入力中（word.html）
SENTENCE_INPUT → 文章入力中（sentence.html）
VOTE_INPUT     → 投票中（vote.html）
```

※ リザルト画面なし。3ラウンド終了後は WAITING に戻り、start画面で前回結果(beforeResult)を表示。

### 遷移図

```
[WAITING] --PUT /rooms (開始)--→ [WORD_INPUT]
[WORD_INPUT]   --全員単語提出--→ [SENTENCE_INPUT]
[SENTENCE_INPUT] --全員文章提出--→ [VOTE_INPUT]
[VOTE_INPUT]   --全員投票--→ [WORD_INPUT] (次のラウンド) or [WAITING] (3ラウンド終了)
```

---

## 3. 状態遷移のトリガー（バックエンド）

### 3.1 WAITING → WORD_INPUT

- **トリガー**: `PUT /rooms/{passphrase}` が呼ばれたとき
- **処理**: `status = "WORD_INPUT"`, `round = 1` に更新
- **冪等**: 既に `WORD_INPUT` 以降なら 200 を返すだけ（何もしない）

### 3.2 WORD_INPUT → SENTENCE_INPUT

- **トリガー**: 単語提出時、**全員分の単語が揃ったとき**
- **判定**: `users` の当該ルーム人数 = `words` の当該ラウンドの投稿者数
- **処理**:
  1. ユーザー4個 + ランダム6個で10個のプールを作成
  2. その中から4個を選び、配布単語として保存
  3. `status = "SENTENCE_INPUT"` に更新

### 3.3 SENTENCE_INPUT → VOTE_INPUT

- **トリガー**: 文章提出時、**全員分の文章が揃ったとき**
- **判定**: `users` の当該ルーム人数 = `sentences` の当該ラウンドの投稿者数
- **処理**: `status = "VOTE_INPUT"` に更新

### 3.4 VOTE_INPUT → WORD_INPUT or WAITING

- **トリガー**: 投票時、**全員分の投票が揃ったとき**
- **判定**: `users` の当該ルーム人数 = `votes` の当該ラウンドの投票者数
- **処理**: 集計なし。`round >= 3` なら `status = "WAITING"`（start画面に戻る）、そうでなければ `round++`, `status = "WORD_INPUT"`

## 4. GET /rooms のレスポンス仕様

クライアントがポーリングで取得するレスポンス。**これだけで画面遷移と表示が決まる**。

```json
{
  "status": "WORD_INPUT",
  "members": [
    {
      "userId": "xxx",
      "name": "jintan",
      "beforeResult": [
        { "sentence": "ふにゃふにゃ砂肝生炒め(激安)", "voteCount": 2 },
        { "sentence": "激安ふにゃふにゃ生卵炒め", "voteCount": 1 }
      ]
    }
  ],
  "goal": "世界で一番食べたくないもの",
  "distributedWords": ["激安", "ふにゃふにゃ", "生", "炒め"]
}
```

| フィールド | 型 | 説明 | いつ入るか |
|------------|-----|------|------------|
| status | string | WAITING/WORD_INPUT/SENTENCE_INPUT/VOTE_INPUT | 常に |
| members | array | 参加者一覧。beforeResultは前ラウンドの文章・得票数 | 常に |
| goal | string | お題 | WORD_INPUT 以降 |
| distributedWords | string[] | 配布された単語 | SENTENCE_INPUT, VOTE_INPUT のとき |

---

## 5. 配布単語の保存（distributed_words）

**仕様書のDB設計に定義済み**。`distributed_words` テーブルを使用。

| カラム | 型 | 説明 |
|--------|-----|------|
| room_passphrase | VARCHAR | PK (複合) |
| round | INT | PK (複合) |
| value | VARCHAR | カンマ区切り "激安,ふにゃふにゃ,生,炒め" |

- 1ラウンド1行
- 取得時は `value.split(",")` で配列化

---

## 6. 全員完了の検知（簡易実装）

各フェーズで、**投稿APIの処理内**でチェックする。

```java
// 例: POST /words の処理の最後
int memberCount = userDao.countByRoomPassphrase(passphrase);
int submittedCount = wordDao.countByPassphraseAndRound(passphrase, round);
if (memberCount == submittedCount) {
    // 単語プール作成・配布・status更新
    distributeWordsAndTransitionToSentence(passphrase, round);
}
```

- `words`, `sentences`, `votes` は **1人1投稿** が前提
- 重複投稿は **上書き** で扱う（同じ user_id + round なら UPDATE）

## 7. フロントエンドのポーリング実装

### 7.1 共通ポーリング関数（1つ作れば全画面で使える）

```javascript
const POLL_INTERVAL_MS = 2000;

function startPolling(callback) {
  const poll = async () => {
    try {
      const room = await API.getRoomStatus();
      callback(room);
    } catch (e) {
      console.error(e);
    }
  };
  poll(); // 即実行
  return setInterval(poll, POLL_INTERVAL_MS);
}

function stopPolling(intervalId) {
  if (intervalId) clearInterval(intervalId);
}
```

### 7.2 画面ごとの使い方

| 画面 | ポーリングで見るもの | 遷移条件 |
|------|---------------------|----------|
| start.html | members, beforeResult, status | status !== "WAITING" → word.html |
| word.html | members, goal, status | status === "SENTENCE_INPUT" → sentence.html |
| sentence.html | goal, distributedWords, status | status === "VOTE_INPUT" → vote.html |
| vote.html | members, goal, status | status === "WORD_INPUT" → word.html / status === "WAITING" → start.html |

### 7.3 遷移の実装パターン

```javascript
let prevStatus = '';
let prevRound = 0;

const id = startPolling((room) => {
  // 参加者リスト更新
  renderMembers(room.members);

  // 状態変化で遷移
  if (prevStatus === 'WAITING' && room.status !== 'WAITING') {
    stopPolling(id);
    location.href = 'word.html';
    return;
  }
  if (prevStatus === 'WORD_INPUT' && room.status === 'SENTENCE_INPUT') {
    stopPolling(id);
    location.href = 'sentence.html';
    return;
  }
  // ...

  prevStatus = room.status;
  if (room.round) prevRound = room.round;
});
```

## 8. 同時開始ボタン（冪等）

- 複数人が同時に `PUT /rooms` を叩いても問題ないようにする
- **方針**: `status` が `WAITING` のときだけ `WORD` に更新
- 既に `WORD` 以降なら、そのまま 200 を返す（DB更新なし）

```java
if ("WAITING".equals(currentStatus)) {
    roomDao.updateStatus(passphrase, "WORD_INPUT");
    roomDao.updateRound(passphrase, 1);
}
return ResponseEntity.ok().build();
```

## 9. 実装順序（7時間想定）

| 順 | 内容 | 目安 |
|----|------|------|
| 1 | DB初期化 + rooms に status 初期値 "WAITING" | 30分 |
| 2 | GET /rooms 実装（status, members(beforeResult含む), goal, distributedWords） | 1時間 |
| 3 | PUT /rooms 実装（WAITING→WORD_INPUT） | 30分 |
| 4 | POST /words 内で全員完了検知 → SENTENCE_INPUT 遷移 | 1時間 |
| 5 | distributed_words テーブル + 配布ロジック | 1時間 |
| 6 | POST /sentences 内で全員完了検知 → VOTE_INPUT 遷移 | 30分 |
| 7 | POST /votes 内で全員完了検知 → WORD_INPUT/WAITING 遷移 | 1時間 |
| 8 | フロント: 共通ポーリング + 各画面の遷移 | 1.5時間 |

---

## 10. 省略してよいもの（7時間枠）

| 項目 | 省略 |
|------|------|
| ポーリングのバックオフ | 2秒固定でOK |
| 離脱検知 | なし（room_users から削除しない） |
| エラー時のリトライ | 次回ポーリングで自然に復帰 |
| ラウンド数の設定 | 3に固定、集計なし |

## 11. チェックリスト

- [ ] rooms.status の初期値は "WAITING"（status値: WAITING/WORD_INPUT/SENTENCE_INPUT/VOTE_INPUT）
- [ ] GET /rooms は passphrase 必須、X-User-Id 必須
- [ ] 全員完了は「人数 == 投稿数」で判定
